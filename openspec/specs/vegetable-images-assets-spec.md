# Spec: Vegetable Image Assets & Data Sync

**Status:** Spec (design and process)  
**Context:** Organize preset images under a dedicated asset folder, define naming and sync with `VEGETABLE_PRESETS`, and support a CLI to optimize images to WebP.

---

## 1. Goals

- **Single place for preset images**: All vegetable preset images live in a well-defined folder, with a clear naming convention.
- **Sync with data**: Image paths in `src/lib/vegetables-data.ts` stay in sync with the asset folder (no orphan files, no missing images for presets).
- **Optimized delivery**: Images are served as WebP for size and quality; a CLI converts source assets to WebP.

---

## 2. Naming Convention

### 2.1 Slug from preset name

Each preset has a **display name** (e.g. `"Bell Pepper"`, `"Green Bean"`, `"Brussels Sprouts"`). The **asset slug** is derived as:

- Lowercase
- Spaces and punctuation replaced with a single hyphen `-`
- No leading/trailing hyphens

**Examples:**

| Preset name      | Asset slug         |
| ---------------- | ------------------ |
| Bell Pepper      | `bell-pepper`      |
| Green Bean       | `green-bean`       |
| Brussels Sprouts | `brussels-sprouts` |
| Sweet Potato     | `sweet-potato`     |
| Chili Pepper     | `chili-pepper`     |

**Rule:** One canonical image per preset. The slug is unique per entry in `VEGETABLE_PRESETS`. If you later add variants (e.g. red vs green bell pepper), use a secondary convention such as `bell-pepper-red`, `bell-pepper-green`; the **default** image for the preset remains `bell-pepper` (or the first variant you define).

### 2.2 File names

- **Source (authoritative):** `{slug}.png` or `{slug}.jpg` — e.g. `bell-pepper.png`. These are the files you edit and commit.
- **Output (served):** `{slug}.webp` — e.g. `bell-pepper.webp`. Generated by the optimize CLI; the app references these URLs.

## 3. Folder Layout

```
public/
  vegetables/           ← served by Vite (URLs: /vegetables/...)
    bell-pepper.webp    ← optimized WebP (CLI output)
    carrot.webp
    ...
  vegetables-source/    ← optional: raw sources (not served, or served only in dev)
    bell-pepper.png
    carrot.jpg
    ...
```

**Variant A (recommended):**

- **Source:** `public/vegetables-source/` — only PNG/JPG sources; not required to be served.
- **Output:** `public/vegetables/` — WebP only; app and Vite serve from here.
- **URL for a preset:** `/vegetables/{slug}.webp`.

**Variant B (single folder):**

- Single folder `public/vegetables/` with both `*.png`/`*.jpg` and `*.webp`.
- CLI overwrites or writes WebP next to sources.
- Slightly messier (mixed extensions); Variant A keeps source and output clearly separate.

**Recommendation:** Use **Variant A**. The app only ever references `/vegetables/{slug}.webp`. If a file is missing, the app falls back to `DEFAULT_IMAGE_URL` (placeholder) as today.

## 4. Syncing Assets ↔ Data

### 4.1 Source of truth

- **Preset list:** `VEGETABLE_PRESETS` in `src/lib/vegetables-data.ts` is the source of truth for _which_ vegetables exist.
- **Image path:** For each preset, `imageUrl` should resolve to the asset for that preset. Two approaches:

**Option 1 — Derived at runtime (recommended)**

- Do not store full URLs in the preset array for local assets.
- Define a small helper, e.g. `getPresetImageUrl(preset: VegetablePreset): string`:
     - Input: preset name → slug (same rule as file naming).
     - Return: `"/vegetables/" + slug + ".webp"`.
- In the UI, use `getPresetImageUrl(preset)` (or equivalent) so that adding a new preset or renaming does not require touching image paths.
- **Fallback:** If you need to support “no image yet” per preset, either (a) check for file existence at build time and keep a `hasImage: boolean` or (b) keep using `DEFAULT_IMAGE_URL` when the derived path 404s (e.g. via img `onError`). Current app already uses `onError` fallback on cards.

**Option 2 — Explicit URL in data**

- Each preset keeps `imageUrl: string` (e.g. `"/vegetables/bell-pepper.webp"`).
- A **sync script** (or CLI step): for each preset, set `imageUrl = "/vegetables/" + slug(name) + ".webp"`.
- Ensures data and folder stay aligned but duplicates the slug logic and requires running the script when presets change.

**Recommendation:** **Option 1** (derived URL from preset name). Single source of truth: preset name → slug → path. No duplicate URLs in the data file; only `DEFAULT_IMAGE_URL` for fallback when the asset is missing.

### 4.2 Validation (optional CLI step)

A **validate** step can:

- List all presets and compute expected slugs.
- Check that `public/vegetables/{slug}.webp` exists for each.
- Optionally list files in `public/vegetables/` that do not match any preset slug (orphans).
- Exit with non-zero if any preset is missing an image (or treat as warning if you allow placeholders).

This keeps “folder ↔ data” sync checkable and scriptable.

## 5. Image Optimization CLI

### 5.1 Role of the CLI

- **Input:** Source images in `public/vegetables-source/` (e.g. `*.png`, `*.jpg`).
- **Output:** WebP files in `public/vegetables/`, one per source file, same base name: `{slug}.webp`.
- **Optional:** Validate that every `VEGETABLE_PRESETS` preset has a corresponding `{slug}.webp` (or report missing).

### 5.2 Tool choice

**Recommended: Sharp (Node.js)**

- **Why:** Single dependency, no system binaries, fast, supports WebP (quality, effort, lossless options). Fits into `npm run optimize-images` or similar.
- **Usage:** Read from `vegetables-source/`, resize if desired (e.g. max width 400), write WebP to `vegetables/`.
- **Alternative:** `imagemin` + `imagemin-webp` — more plugins, slightly heavier.
- **Alternative:** `cwebp` (Google) — CLI binary, need to install on each machine or in CI.

**Recommendation:** Use **Sharp** in a small Node script (e.g. `scripts/optimize-vegetable-images.ts` or `.js`) so the repo stays self-contained and CI-friendly.

### 5.3 Suggested script behavior

1. Ensure `public/vegetables/` and optionally `public/vegetables-source/` exist.
2. For each file in `vegetables-source/` with extension `.png` or `.jpg` (and optionally `.jpeg`):
      - Base name = slug (file name without extension).
      - Encode to WebP (e.g. quality 85, or lossless for graphics with sharp edges).
      - Write to `public/vegetables/{baseName}.webp`.
3. Optional: Read `VEGETABLE_PRESETS`, compute slugs, report missing WebP files.
4. Optional: List orphan WebPs (no matching preset).

## 6. Bell Pepper Example (your first image)

- **Preset:** `"Bell Pepper"` in `VEGETABLE_PRESETS`.
- **Slug:** `bell-pepper`.
- **Current file:** `public/ChatGPT Image Mar 4, 2026, 11_53_00 PM.png` (red bell pepper).
- **Target:**
     - Move/copy to `public/vegetables-source/bell-pepper.png` (or keep as `.jpg` and name `bell-pepper.jpg`).
     - Run optimize CLI → `public/vegetables/bell-pepper.webp`.
- **App:** With Option 1 (derived URL), the app would resolve `"Bell Pepper"` → `"/vegetables/bell-pepper.webp"` and use that; no change to the preset object except possibly removing a hardcoded `imageUrl` in favor of the helper.

## 7. Summary

| Item              | Decision                                                                                  |
| ----------------- | ----------------------------------------------------------------------------------------- |
| Slug              | Lowercase, spaces → hyphens, e.g. `bell-pepper`, `green-bean`.                            |
| Source folder     | `public/vegetables-source/` — raw PNG/JPG.                                                |
| Output folder     | `public/vegetables/` — WebP only, served at `/vegetables/*.webp`.                         |
| Data sync         | Derive image URL from preset name (slug + path); fallback to placeholder if file missing. |
| Optimization tool | Sharp (Node) in a script; `npm run optimize-images`.                                      |
| Validation        | Optional CLI step: ensure each preset has `{slug}.webp`; report orphans.                  |

This spec keeps the asset folder organized, naming consistent, and the data model in sync with the folder while leaving implementation to a follow-up change (e.g. `/opsx:new` or `/opsx:ff`).
